How to Ask Me for Help

Always welcome

What to Include

When asking for help, please include these four elements. This saves everyone time and leads to better solutions.

  • Context - What you're trying to accomplish
  • Attempts - What you've already tried
  • Evidence - Errors, logs, or screenshots
  • Goal - What "done" looks like

"Hey, I'm trying to deploy the API to staging but getting a 502 error. I've checked the logs and restarted the service twice. Here's the error from the load balancer logs: [screenshot]. I need the API to return 200 OK on the health endpoint."

"Hey, the deployment is broken. Can you help?"

Pro Tip

This approach makes you look senior and saves everyone time. For more on asking smart questions, read Eric Raymond's "How To Ask Questions The Smart Way" or my friendlier version.

Async-First Work Style

Async by default

My Default Mode

  • I work async-first by default
  • I usually respond within 4-8 hours on weekdays
  • Weekends are for rest and recovery
  • Deep work blocks are protected time

Urgent Matters

If something is truly urgent:

  • Start message with [URGENT]
  • Text my phone (for critical production issues only)
  • Don't send "hello" and wait for response
  • Include all context in first message

Response Times

4 hours Non-blocking questions
1 business day Code reviews, PR feedback
2-3 days Feature requests, architectural discussions

Meeting Philosophy

When Meetings Are Worth It

  • Complex architectural decisions
  • Sensitive team discussions
  • Brainstorming sessions (with clear goals)
  • Relationship building with new partners
  • Retrospectives and post-mortems

Meeting Requirements

  • Agenda-first: Send agenda 24h in advance
  • Time-boxed: Default to 25 or 50 minutes
  • Documented: Notes and action items shared after
  • Optional: Recorded for those who can't attend

No Meeting If...

We shouldn't have a meeting if this can be a document, a Slack thread, or a Loom video. This attracts people who value focus and respects everyone's time.

Code Review Philosophy

What I Focus On

  • Readability: Can others understand this?
  • Correctness: Does it work as intended?
  • Edge cases: What happens when things fail?
  • Security: Any obvious vulnerabilities?
  • Performance: Any obvious bottlenecks?

What I Don't Nitpick

  • Personal style preferences
  • Minor refactors without clear benefit
  • Exact variable naming (within reason)
  • Perfect test coverage on prototype code
  • Architecture on small, temporary changes

Feedback Delivery

  • Questions over statements: "What if...?" not "You should..."
  • Positive reinforcement for good patterns
  • Specific, actionable suggestions
  • Context about why, not just what
  • Assumes positive intent

PR Preferences

  • Small PRs: ~300 lines max when possible
  • Clear descriptions: What and why
  • Testing instructions: How to verify
  • Screenshots: For UI changes
  • Self-contained: One logical change per PR

Asking Smart Questions (Friendly Version)

1. Do Your Homework First

Check documentation, search existing issues, try basic troubleshooting.

2. Be Specific

"The login page shows error 500" is better than "it's broken."

3. Share What You've Tried

This shows effort and helps avoid suggesting solutions you've already attempted.

4. Include Relevant Details

Version numbers, environment, error messages, screenshots.

5. Ask, Don't Demand

"Could you help me understand..." vs "Fix this for me."
